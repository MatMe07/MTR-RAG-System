# Задача 2. Первый экран эксперта на тестовых данных

## Цель

Собрать видимую демонстрацию основного сценария MVP:

1. Эксперт вводит запрос или выбирает паспорт.
2. Система показывает извлеченную карточку изделия.
3. Система выводит ранжированный список кандидатов.
4. Эксперт видит совпадения, расхождения, предупреждения и источники.
5. Эксперт подтверждает результат, отклоняет его или отмечает необходимость проверки.

На этом этапе экран работает на подготовленных тестовых данных. Готовый backend ждать не нужно. Позже фиктивные данные будут заменены вызовом API напарника.

## Рекомендуемый инструмент

Для первого MVP использовать Streamlit: он позволяет быстро собрать интерфейс на Python и нормально отображает русский текст. Если позднее команда выберет React, сценарий и структура экрана сохранятся.

## Что сделать

### 1. Описать экран

Создать `docs/ui/expert_search_screen.md` и перечислить блоки экрана:

- выбор режима поиска;
- поле запроса и загрузка паспорта;
- кнопка «Найти кандидатов»;
- редактируемая карточка изделия;
- таблица Top-20;
- подробности выбранного кандидата;
- совпавшие, отличающиеся и отсутствующие параметры;
- предупреждения по H2S, CO2, покрытию, давлению и стандарту;
- источники с файлом, страницей и фрагментом;
- решение эксперта и поле комментария.

### 2. Подготовить демонстрационный ответ

Создать `data/sample/ui_demo_case_q007.json` для запроса Q007. В ответе должны быть:

- исходный запрос;
- карточка требуемого отвода;
- минимум пять кандидатов;
- первым кандидатом MTR-0013;
- кандидат MTR-0001 с предупреждениями по H2S и внутреннему покрытию;
- кандидат MTR-0012 с отличием класса прочности;
- кандидат MTR-0014 с отличием толщины стенки;
- кандидат MTR-0002 с отличием угла;
- ссылки на тестовые паспорта и страницы.

Структуру кандидатов брать из `docs/schemas/match_result.schema.json`.

### 3. Собрать интерфейс

Создать `frontend/app.py`. Первый вариант может читать `ui_demo_case_q007.json` с диска.

Экран должен позволять:

- ввести или изменить запрос;
- нажать кнопку поиска;
- увидеть карточку запроса;
- увидеть таблицу кандидатов;
- выбрать кандидата и раскрыть объяснение;
- увидеть предупреждения заметно, но без автоматического удаления кандидата;
- записать решение эксперта: «подтвердить», «отклонить», «требует проверки»;
- оставить текстовый комментарий.

### 4. Подготовить подключение backend

В коде отделить загрузку результатов в одну функцию. Сейчас она читает JSON-файл, позже внутри нее появится запрос к API. Формат входа и результата согласовать с напарником до подключения.

## Что проверить вручную

- Русские буквы отображаются корректно.
- MTR-0013 находится на первом месте в сценарии Q007.
- Покрытие и H2S показаны как предупреждения или подтвержденные свойства, а не как скрытый автоматический отказ.
- У каждого объяснения есть понятный источник.
- Длинные названия и комментарии не обрезаются.
- Интерфейс не утверждает, что система приняла инженерное решение вместо эксперта.

## Критерий готовности

Задача готова, когда экран запускается локально без backend, демонстрирует Q007 от запроса до решения эксперта и использует структуру `MatchResult` без добавления несовместимых полей.

## Что согласовать с напарником

- как будет называться endpoint поиска;
- в каком JSON backend принимает карточку запроса;
- возвращает ли backend список `MatchResult` напрямую;
- как передаются имя файла, номер страницы и исходный фрагмент;
- куда позднее сохранять решение и комментарий эксперта.
