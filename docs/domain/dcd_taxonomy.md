# DCD-разметка тестового корпуса

## Что означает DCD

DCD — это три уровня контекста:

1. `domain` — общая предметная область;
2. `collection` — однородный класс изделий или источников;
3. `document` — конкретный паспорт, OCR-файл, каталог или регламент.

Для MVP используется один домен `gas_pipeline_mtr` — «МТР газопровода».
Единые коды коллекций описаны в `dcd_taxonomy.json`.

## Основные классы

| Код коллекции | Класс | Ключевые характеристики |
|---|---|---|
| `pipes` | трубы | DN, толщина стенки, материал, класс прочности, PN, среда, покрытие, ГОСТ/ТУ |
| `elbows` | отводы | подтип, DN, угол, толщина стенки, материал, класс прочности, PN, среда, покрытие, ГОСТ/ТУ |
| `reducers` | переходы | D1, D2, исполнение, толщина стенки, материал, PN, ГОСТ/ТУ |
| `valves` | арматура | вид арматуры, DN, PN, привод, материал, класс герметичности, среда, ГОСТ/ТУ |
| `plugs` | заглушки | тип заглушки, DN, PN, материал, покрытие, ГОСТ/ТУ |

`tees` оставлена вспомогательной коллекцией, потому что тройник уже входит в
тестовый газопровод и корпус документов. `regulations` хранит ЛНД и другие
источники правил; это не класс изделия.

## Правила разметки

- Один паспорт относится к одной коллекции по изделию, которое он описывает.
- OCR-текст сохраняет коллекцию исходного паспорта, но получает собственный
  `document_id` и тип `ocr_text`.
- ЛНД относится к `regulations`. Факты из него не становятся характеристиками
  конкретного изделия без ссылки на правило или экспертного подтверждения.
- Каталожная карточка и паспорт одного изделия могут иметь разные
  `document_id`. Их связь выполняется через код МТР/КСМ и подтверждённые факты.
- Для каждого факта сохраняется конкретный `source_fragment`; одного имени
  документа недостаточно для проверяемого объяснения.

## Где используется

DCD ограничивает область поиска до нужного класса, помогает не смешивать трубу
с отводом и даёт агенту список коллекций и документов, которые разрешено
проверять. Разметка тестовых файлов находится в
`data/sample/document_manifest.csv`.
