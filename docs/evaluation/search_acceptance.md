# Проверка поиска и объяснений

## Эталонные данные

`data/sample/golden_dataset.csv` содержит 30 кейсов с правильной позицией МТР,
статусом, предупреждением или уточняющим вопросом.

`data/evaluation/golden_assertions.jsonl` дополняет самые рискованные кейсы:

- точные коды МТР и КСМ;
- ошибки OCR;
- расхождения DN, PN, толщины, угла и класса прочности;
- неподтверждённые H2S, CO2 и покрытие;
- составные замены `2 x 45°` и `3 x 30°`;
- обязательные источники и содержание объяснения.

## Формат фактических результатов

Поисковый прогон должен сохранить по одной JSON-строке на кейс:

```json
{
  "case_id": "Q025",
  "ranked_mtr_codes": ["MTR-0001", "MTR-0013"],
  "warnings": ["Пригодность к H2S не подтверждена"],
  "explanation": "Нужно проверить паспорт и применимый ЛНД.",
  "sources": [
    "documents/passport_001_oksh90_dn159.md",
    "documents/lnd_extract.md"
  ]
}
```

## Метрики

- `Top-3 Hit Rate` — доля кейсов, где правильный МТР входит в первые 3.
- `Top-20 Hit Rate` — доля кейсов, где правильный МТР входит в первые 20.
- `Exact Code Top-1` — доля точных запросов, где код стоит первым.
- `Warning Coverage` — все обязательные части предупреждения присутствуют.
- `Explanation Coverage` — объяснение содержит проверяемые причины.
- `Source Coverage` — перечислены все обязательные документы.

Порог `Top-3 >= 70%` и `Exact Code Top-1 >= 95%` взят из ТЗ.
Для `Top-20` задан рабочий порог `95%`, потому что система должна сокращать
выдачу до 15–20 кандидатов; этот порог нужно отдельно утвердить у куратора.
Покрытие обязательных предупреждений, объяснений и источников должно быть 100%.

## Запуск

```powershell
python -m backend.app.scripts.evaluate_search_results --results data/evaluation/actual_search_results.jsonl --output data/generated/evaluation_report.json
```

Оценщик не подменяет результаты ожидаемыми ответами. Если кейс отсутствует в
файле фактического прогона, он считается не пройденным.
