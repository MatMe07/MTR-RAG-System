# Маршрутизация обычного и агентного поиска

## Зачем нужен маршрутизатор

Не каждый запрос стоит отправлять в тяжёлый агентный цикл. Точный код или
полностью заполненная карточка быстрее и надёжнее обрабатываются обычным
поиском. Агент нужен, когда ответ приходится собирать по нескольким коллекциям,
документам или правилам.

## Обычный поиск

Маршрут `ordinary` используется, если достаточно одного шага:

- точный поиск по коду МТР, КСМ или обозначению;
- фильтрация по заполненным характеристикам одного класса;
- семантический или гибридный поиск кандидатов в одной коллекции;
- поиск по одному паспорту после извлечения карточки;
- повторный поиск после ручного исправления карточки экспертом.

Обычный маршрут может включать PostgreSQL, полнотекстовый поиск, Qdrant и rules
engine. Слово «обычный» означает отсутствие многошагового агента, а не только
один технический индекс.

## Уточнение

Маршрут `clarification` используется до поиска, когда отсутствуют параметры,
без которых выдача будет вводить эксперта в заблуждение. Например, для отвода
известны только тип и DN, но не заданы угол, толщина и материал.

Уточнение не является агентным поиском. Сначала система задаёт короткий вопрос,
после ответа снова выбирает маршрут.

## Агентный поиск

Маршрут `agent` нужен, если присутствует хотя бы один признак:

- необходимо собрать совместимый набор изделий для объекта газопровода;
- требуются доказательства из паспорта, каталога и ЛНД одновременно;
- источники противоречат друг другу;
- проверяется составная замена, например `2 x 45°` вместо `1 x 90°`;
- надо последовательно применить несколько правил и показать их источники;
- запрос затрагивает несколько DCD-коллекций.

Агент не утверждает аналог автоматически. Его результат — ранжированные
кандидаты, evidence-фрагменты, предупреждения и вопросы эксперту.

## Порядок обработки

1. Из запроса или паспорта строится карточка.
2. Проверяется полнота ключевых параметров.
3. Маршрутизатор выбирает `ordinary`, `clarification` или `agent`.
4. Обычный поиск возвращает Top-N и передаёт кандидатов в rules engine.
5. Агент выполняет несколько поисковых шагов, сохраняет evidence и только затем
   передаёт кандидатов в rules engine.
6. Эксперт видит причины попадания в выдачу, предупреждения и источники.

Машинная реализация политики находится в
`backend/app/services/search_router.py`, а контрольные сценарии — в
`data/evaluation/complex_queries.jsonl`.
