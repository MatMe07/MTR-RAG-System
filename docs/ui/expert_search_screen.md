# Экран экспертного поиска

## Назначение

Экран помогает эксперту сократить большой каталог до списка наиболее близких кандидатов. Система показывает причины попадания каждого кандидата в выдачу, но не принимает окончательное инженерное решение вместо эксперта.

Первый демонстрационный сценарий использует запрос Q007 и локальный файл `data/sample/ui_demo_case_q007.json`. После подключения backend источник данных изменится, а структура экрана и формат результатов останутся прежними.

## Основной сценарий

1. Эксперт выбирает режим поиска.
2. Эксперт вводит текст запроса или прикладывает паспорт.
3. Система формирует карточку требуемого изделия.
4. Эксперт при необходимости исправляет распознанные параметры.
5. Система показывает до 20 кандидатов в порядке релевантности.
6. Эксперт открывает кандидата и проверяет совпадения, расхождения, предупреждения и источники.
7. Эксперт подтверждает кандидата, отклоняет его или отправляет на дополнительную проверку и оставляет комментарий.

## Блоки экрана

### 1. Панель запроса

- выбор режима: точный, семантический, гибридный, поиск по паспорту или проверка аналога;
- многострочное поле запроса;
- загрузка паспорта в PDF, PNG, JPG или TXT;
- кнопка `Найти кандидатов`;
- пояснение, что в демонстрации используется подготовленный ответ Q007.

### 2. Карточка требуемого изделия

Карточка редактируется экспертом до повторного поиска. На экране показываются:

- тип и подтип изделия;
- DN, угол и толщина стенки;
- PN;
- марка стали и класс прочности;
- рабочая среда и подтверждение H2S/CO2;
- внутреннее и наружное покрытие;
- ГОСТ/ТУ;
- источник и уверенность извлечения.

Неизвестное значение отображается как `не указано`, а не как отрицательное подтверждение.

### 3. Таблица кандидатов Top-20

Для каждого кандидата таблица показывает:

- место в выдаче;
- код МТР и код КСМ;
- наименование;
- рекомендательный статус;
- процент близости;
- количество предупреждений.

Процент близости служит для ранжирования и не означает автоматического допуска изделия к эксплуатации.

### 4. Подробности кандидата

Эксперт выбирает кандидата из списка и видит:

- совпавшие параметры;
- отличающиеся параметры;
- параметры, которых нет в источниках;
- предупреждения по H2S, CO2, покрытию, давлению, материалу и ГОСТ/ТУ;
- краткий комментарий системы;
- сработавшие правила и их реакцию;
- источники: тип, имя файла, страница или строка и исходный фрагмент.

Предупреждение не удаляет кандидата из выдачи. Например, отсутствие подтверждения внутреннего покрытия или H2S переводит результат в статус `требует проверки`.

### 5. Решение эксперта

Эксперт выбирает одно решение:

- `подтвердить`;
- `отклонить`;
- `требует проверки`.

Рядом находится обязательный для спорных случаев текстовый комментарий. В демонстрации решение хранится только в состоянии открытого приложения; после подключения backend оно должно записываться в таблицу экспертных решений и журнал аудита.

## Состояния экрана

- `готов к поиску`: запрос можно изменить;
- `идет поиск`: кнопка блокируется, показывается индикатор;
- `результаты найдены`: отображаются карточка и кандидаты;
- `ничего не найдено`: предлагается уточнить параметры;
- `ошибка`: показывается понятное сообщение без технической трассировки;
- `решение записано`: показывается выбранный статус и кандидат.

## Будущий контракт с backend

Функция интерфейса `load_search_results` должна быть единственной точкой получения поисковой выдачи. Сейчас она читает JSON с диска. Позднее она отправит в backend режим поиска, текст запроса, карточку изделия и метаданные файла, а в ответ получит список объектов `MatchResult`.

Предлагаемые методы API:

- `POST /api/v1/search` — построить карточку запроса и вернуть кандидатов;
- `POST /api/v1/expert-decisions` — сохранить решение эксперта и комментарий.

Для источника backend передает `type`, `file`, `page`, `row` и `fragment`. Кандидаты должны соответствовать `docs/schemas/match_result.schema.json`.
